# Feature Specification: デジタル庁DS準拠 16:9 プレゼンテンプレート

**Feature Branch**: `001-ds-presentation-template`

**Created**: 2026-07-05

**Status**: Draft

**Input**: User description: "@docs/requirements.md — デジタル庁デザインシステムを厳守した 16:9 プレゼンテンプレート。デジタル庁ホームページ配色を踏襲。プレゼン/データ可視化/認知・行動心理のベストプラクティスを引用付きで埋め込み、ビジネス・ソフトウェア開発で実用。DS 更新を低コストで取り込み、将来は編集可能な PowerPoint へ変換。"

## Clarifications

### Session 2026-07-05

- Q: 図表レイアウトの提供範囲(実描画チャートか作図枠か) → A: 両方 — トークン整形済みの静的サンプル図(主要数種)を規範として同梱し、作成者の実データ図は画像/プレースホルダ枠で受ける。
- Q: 日本語フォントの同梱方針 → A: 既定は非同梱(システム + Noto Sans JP フォールバック)。ピクセル忠実/オフライン完全再現向けに同梱手順をオプションとして文書化。
- Q: デザイントークンの固定バージョン → A: 実測済み最新 v2.0.1 を厳密固定。更新は明示操作(バージョン更新→差分レビュー→視覚回帰)として扱う。
- Q: 注釈レイヤーの既定表示状態 → A: 既定は非表示(発表用クリーン)、トグルで表示。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 準拠デックを素早く組み立てる (Priority: P1)

ビジネス/ソフトウェア開発の担当者が、コアの 8 レイアウト(表紙・目次・章扉・本文・2カラム比較・図表・まとめ・参考)を選んで内容を差し込むだけで、デジタル庁デザインシステムに準拠した 16:9 のプレゼン一式を作成できる。ブランド配色・タイポ・余白の判断に迷わず、体裁が自動的に整う。

**Why this priority**: これがテンプレートの中核価値(準拠と再利用性)。この 1 本だけでも「準拠デックを短時間で作る」という MVP を単独提供できる。

**Independent Test**: 8 レイアウトを使って 10 枚程度のデックを組み、16:9 で崩れず表示・PDF 出力できること、配色・タイポがブランド基準どおりであることを確認すれば単独で検証できる。

**Acceptance Scenarios**:

1. **Given** 空のテンプレート, **When** 8 レイアウトのそれぞれに見出しと本文を入力する, **Then** すべてのスライドが 16:9 で崩れず(はみ出し・見切れなし)表示される。
2. **Given** 作成したデック, **When** 印刷/PDF 出力する, **Then** 1 スライド = 1 ページで出力される。
3. **Given** 任意のスライド, **When** 配色・文字色・余白を確認する, **Then** すべてがブランド基準(デザイントークン)由来で、独自の色値が混在しない。

---

### User Story 2 - 実証されたベストプラクティスを引用付きで適用する (Priority: P2)

作成者(および学習者)が、各スライドに埋め込まれた設計意図の注釈と巻末の出典一覧を通じて、構成・スライド設計・データ可視化・認知/行動心理の確立された手法を、根拠付きで理解・適用できる。手法は解説ドキュメントにも出典付きで列挙される。

**Why this priority**: テンプレートを「体裁が整うだけ」から「効果的な伝達を再現できる」教材兼実務ツールへ引き上げる差別化価値。P1 の上に積む増分。

**Independent Test**: 各スライドの注釈をトグルで表示し、適用手法と出典が示され、巻末リファレンスおよび解説ドキュメントと相互参照できることを確認すれば単独で検証できる。

**Acceptance Scenarios**:

1. **Given** 任意のスライド, **When** 注釈レイヤーを表示に切り替える, **Then** そのスライドで適用している手法と出典が表示される。
2. **Given** 注釈レイヤー, **When** 非表示に切り替える, **Then** 発表用のクリーンな見た目になる。
3. **Given** 解説ドキュメント, **When** 採用手法を確認する, **Then** すべての手法が出典付きで列挙され、テンプレート内リファレンスと相互参照できる。
4. **Given** 説得系の手法(社会的証明・希少性等), **When** テンプレートで用いる, **Then** 事実に基づく強調に限定され、誇張・誘導の用途では提示されない。

---

### User Story 3 - デザインシステム更新を低コストで取り込む (Priority: P2)

テンプレート保守担当が、デジタル庁デザインシステムの更新(色・タイポ・角丸等の変更)を、各レイアウト定義に手を入れることなく、トークン写像層の差分だけで取り込める。更新後も全レイアウトの体裁が保たれる。

**Why this priority**: 準拠を「一度きり」でなく「継続的」に保つための持続性要件。長期の実用性を担保する増分。

**Independent Test**: デザインシステムのバージョンを更新し、レイアウト定義を変更せずに写像層のみの調整で反映できること、更新後に全レイアウトが視覚回帰で許容範囲に収まることを確認すれば単独で検証できる。

**Acceptance Scenarios**:

1. **Given** 新しいデザインシステム版, **When** トークンを更新する, **Then** レイアウト定義を編集せずに写像層のみで反映できる。
2. **Given** トークンの改名・廃止を含む更新, **When** 取り込む, **Then** 写像層のエイリアスが差分を吸収し、テンプレート本体は改修不要である。
3. **Given** 更新適用後, **When** 全レイアウトを更新前と比較する, **Then** 意図しない体裁崩れが無い(視覚回帰が許容範囲)。

---

### Edge Cases

- 見出しや本文が想定量を超えて長い場合、レイアウトはどう振る舞うか(見切れ防止・可読性維持の扱い)。
- デザインシステム更新でトークンが改名・廃止された場合、写像層でどのように吸収し、レイアウトを不変に保つか。
- 標準フォントが利用できない環境(オフライン等)で、代替フォントでも 16:9 の体裁と可読性が保たれるか。
- 図表スライドで系列・要素が多すぎる場合、データ可視化の指針(要素削減・チャート選択)がどう機能するか。
- PDF/印刷出力時に、画面と同等のコントラスト・配色が保たれるか。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: テンプレートは、コア 8 レイアウト(表紙・目次・章扉・本文・2カラム比較・図表・まとめ・参考)を提供しなければならない(MUST)。
- **FR-002**: すべてのレイアウトは 16:9 の標準キャンバスで、はみ出し・見切れなく安定して表示されなければならない(MUST)。
- **FR-003**: 配色・タイポグラフィ・余白・角丸・陰影のすべての視覚スタイルは、デジタル庁デザインシステムが公開するデザイントークンに由来しなければならず、独自のブランド値を持ってはならない(MUST)。
- **FR-004**: 配色はデジタル庁の公開ウェブサイトと整合し、同一のブランドキー色を用いなければならない(MUST)。
- **FR-005**: 各スライドは、設計意図と適用したベストプラクティスを示す注釈レイヤーを備え、表示/非表示を切り替えられなければならない(MUST)。既定は非表示(発表用クリーン)とし、トグルで表示に切り替える。
- **FR-006**: 内容を持つレイアウトの見出しは、アクションタイトル(結論を文で述べる)形式でなければならない(MUST)。
- **FR-007**: 付随する解説ドキュメントは、採用したすべてのベストプラクティスを出典付きで列挙し、テンプレートの参考(リファレンス)スライドと相互参照できなければならない(MUST)。
- **FR-008**: テンプレートは、5 群(構成・ストーリー / スライド設計 / データ可視化 / 認知科学 / 行動心理・意思決定)にわたる採用手法を埋め込み、出典を明示しなければならない(MUST)。
- **FR-009**: テンプレートは、1 スライド = 1 ページの印刷/PDF 出力に対応しなければならない(MUST)。
- **FR-010**: 主要テキストのコントラストは WCAG 2.2 AA を満たさなければならない(MUST)。
- **FR-011**: デザインシステムの更新は、個々のレイアウト定義を編集することなく、トークン写像層の変更のみで取り込めなければならない(MUST)。
- **FR-012**: テンプレートの構造は、編集可能な PowerPoint テンプレートへ変換可能な状態を保たなければならない(固定グリッド・命名レイアウト・標準スライド寸法に整合し、変換を妨げる構成に依存しない)(MUST)。
- **FR-013**: 参考(巻末)スライドは、デック全体で用いたすべての出典を列挙しなければならない(MUST)。
- **FR-014**: 説得志向の手法は、事実に基づく強調のためにのみ適用し、誇張や誘導的操作に用いてはならない(MUST)。
- **FR-015**: 図表レイアウトは、デザイントークンで整形された静的サンプル図(棒・折れ線・構成比の3種)を規範として提供し、かつ作成者の実データ図を受ける画像/プレースホルダ枠を備えなければならない(MUST)。実データのバインディングや作図エンジンは提供範囲に含まない。

### Key Entities *(include if feature involves data)*

- **スライドレイアウト**: 8 種のうちの 1 つ。役割(表紙/目次/…/参考)、グリッド配置、見出し様式、含みうる要素(本文・図表・比較列)を持つ。
- **デザイントークン**: デザインシステム由来の視覚設計値(色・タイポ・余白・角丸・陰影)。テンプレートはこれを参照し、値を複製しない。
- **トークン写像層**: デザイントークンをスライド用の役割名(背景・サーフェス・本文文字色・アクセント等)とスペーシングスケールへ写像する層。更新吸収の窓口。
- **ベストプラクティス**: 採用手法。名称・適用方法・出典・所属群(A〜E)を持つ。
- **注釈**: スライド単位の設計意図解説。適用手法・出典への参照を持ち、表示/非表示を切替可能。
- **出典/リファレンス**: 引用文献の識別情報。注釈・解説ドキュメント・巻末スライドから参照される。
- **デック**: レイアウトを並べた 1 つのプレゼンテーション。全体構造(導入・本体・まとめ・参考)と道標(現在地表示)を持つ。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 8 レイアウトの 100% が 16:9 で見切れなく表示され、1 スライド = 1 ページで PDF 出力できる。
- **SC-002**: 配色・タイポ・余白の値の 100% がデザイントークンに追跡でき、独自ブランド値(ハードコード)が 0 件である。
- **SC-003**: 8/8 のスライドで注釈レイヤーの表示/非表示を切り替えられる。
- **SC-004**: 主要テキストの 100% が WCAG 2.2 AA のコントラストを満たす。
- **SC-005**: 採用手法の 100% が解説ドキュメントに出典付きで掲載され、テンプレートの参考スライドから相互参照できる。
- **SC-006**: 内容を持つレイアウトの見出しが 100% アクションタイトル形式である。
- **SC-007**: デザインシステムのバージョン更新を、レイアウト定義の編集 0 件(写像層のみの変更)で取り込めることを、実際のバージョン更新で確認できる。
- **SC-008**: 配色がデジタル庁ウェブサイトのブランドキー色と一致する。
- **SC-009**: 作成者がレイアウトを用いて、準拠した 10 枚規模のデックを 30 分以内に組み立てられる。
- **SC-010**: 新規の作成者が、埋め込み注釈のみを手がかりに、各スライドで適用された手法を 90% 以上のスライドで正しく識別できる。

## Assumptions

- 内容テキストは日本語を既定とする。標準キャンバスは 16:9(1280×720)で、標準的なプレゼンテーション寸法に整合する。
- デザイントークンはデジタル庁が公開する配布物のバージョン v2.0.1 を厳密固定して参照する。更新は明示操作(バージョン更新→差分レビュー→視覚回帰)として扱う。
- 標準フォントは既定で同梱せず、システムフォント + 標準日本語フォント(Noto Sans JP)のフォールバックとする(可搬性重視・外部依存なし)。ピクセル忠実やオフライン完全再現が必要な配布向けに、同梱手順をオプションとして文書化する。
- ウェブサイト固有の配色差分がある場合は、薄い「ブランドオーバーレイ」層で吸収する(差分の有無は実装時に実サイトで確認)。
- 編集可能な PowerPoint への実変換は後続フェーズで実施する。本フィーチャーでは変換可能性(構造適合)を維持することのみを対象とする。
- ベストプラクティスの一次出典の細部(ページ番号等)は、解説ドキュメント作成時に確認する。

## Out of Scope

- 編集可能な PowerPoint への実変換(後続フェーズ)。
- アニメーション/画面遷移効果。
- CMS 連携。
- 多言語切替。
- ダークテーマ・キーボード操作(任意拡張、本フィーチャーの必須範囲外)。
