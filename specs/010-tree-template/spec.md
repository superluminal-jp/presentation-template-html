# Feature Specification: ツリー構造テンプレート(縦・横)

**Feature Branch**: `010-tree-template`

**Created**: 2026-07-17

**Status**: Draft

**Input**: User description: "ツリー構造のテンプレートを追加。縦と横の両方"

## Overview

デッキに「ツリー構造」を表現するスライドテンプレートを追加する。ツリーが表す内容モデルは
**バーバラ・ミントのピラミッド・プリンシプル**とする — 頂点に主張(結論=支配的メッセージ)、
その下に MECE にグルーピングされた根拠、末端に個別の裏付け/データを置く階層。用途は縦(トップダウン)と
横(左→右)の両方向。縦はコアの新レイアウト、横は付録スライドとして、単一の再利用可能な `.tree`
コンポーネントを両向きで実演する(タイムラインがコア＋付録2段で再利用されるのと同じ方針)。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 縦ツリー(トップダウンの主張構造) (Priority: P1)

発表者が、結論を頂点に置き、その下に MECE な根拠グループ、さらに末端に裏付けを配置した
トップダウンのピラミッド構造を1枚のスライドで示したい。読み手は上から下へ「結論→なぜ→証拠」の
順で論理を追える。

**Why this priority**: ピラミッド・プリンシプルの標準表現はトップダウンであり、デッキの設計思想
(結論先出し・MECE グルーピング)と直結する中核価値。これ単体で「論理構造を1枚で示す」MVP が成立する。

**Independent Test**: 縦ツリーのコアスライドを表示し、頂点ノード1・第2階層の複数ノード・末端ノードが
親子コネクタで結ばれ、1280×720 内に見切れなく収まることを確認できる。

**Acceptance Scenarios**:

1. **Given** デッキを表示した状態, **When** 縦ツリーのスライド(`data-layout="tree"`)へ進む, **Then** 頂点(結論)ノードが最上段、その下に MECE な第2階層ノード群、必要に応じ末端ノードが親子関係のコネクタで結ばれて表示される。
2. **Given** 縦ツリーのスライド, **When** スライドを画面/PDF で確認する, **Then** すべてのノードとコネクタが 1280×720 のスライド枠内に収まり、テキスト・図形の見切れ・はみ出し・重なりがない。
3. **Given** 縦ツリーのスライド, **When** ノードの配色・余白・角丸・文字サイズを確認する, **Then** すべて DS のトークン(役割変数)由来で、生のカラーコードや生の px 余白を含まない。

---

### User Story 2 - 横ツリー(左→右のロジックツリー) (Priority: P2)

発表者が、深い/枝の多いロジックツリーを、トップダウンでは収まらない場合に左→右方向で示したい。
根(結論)を左端に置き、右へ枝を展開する。読み手は左から右へ階層を追える。

**Why this priority**: 縦が収まらない長い/深い構造の受け皿。縦(P1)と同じ `.tree` コンポーネントを
向き替えで再利用するため、P1 の後に低コストで価値を追加できる。付録スライドとして実演する。

**Independent Test**: 付録の横ツリースライドを表示し、根が左端・枝が右へ展開し、コネクタが親子を
結んで 1280×720 内に見切れなく収まることを、縦ツリーと独立に確認できる。

**Acceptance Scenarios**:

1. **Given** デッキの付録区間, **When** 横ツリーのスライド(付録)へ進む, **Then** 根ノードが左、子ノードが右方向に階層展開され、親子がコネクタで結ばれて表示される。
2. **Given** 横ツリーのスライド, **When** 画面/PDF で確認する, **Then** すべてのノードとコネクタが 1280×720 内に収まり、見切れ・はみ出し・重なりがない。
3. **Given** 縦ツリーと横ツリーの両スライド, **When** 双方を比較する, **Then** ノード・コネクタ・状態表現が同一の視覚言語(同じコンポーネント)で、向きだけが異なる。

---

### User Story 3 - 再利用可能なツリーコンポーネント (Priority: P3)

テンプレート利用者(スライド作成者)が、ツリーを他スライドでも再利用でき、階層の深さ・枝数を
差し替えても構造が破綻しないことを期待する。

**Why this priority**: テンプレートとしての一般化価値。P1/P2 の実演が成立していれば付加的。

**Independent Test**: サンプルのノード数/深さを増減しても、レイアウトが崩れず 1280×720 内に収まる
ことを確認できる。

**Acceptance Scenarios**:

1. **Given** ツリーコンポーネント, **When** 第2階層のノード数を増減する, **Then** ノードが均等に配置され、コネクタの親子関係が保たれる。
2. **Given** ツリーコンポーネント, **When** 末端(第3階層)の有無を切り替える, **Then** 階層が2段でも3段でも破綻せず表示される。

---

### Edge Cases

- 第2階層のノードが多い(例: 5枝以上)場合、横幅内に収めつつ可読なノード幅を保てるか。
- ノード本文が長い場合、ノードが縦に伸びてもスライド枠内に収まるか(見切れ防止)。
- 縦ツリーで末端(第3階層)を全ノードに付けた場合、下端がスライド枠を超えないか。
- 濃色スライド系の反転規則と干渉しないか(ツリーは通常スライド前提)。
- 印刷(PDF)時に 1スライド=1ページのページ規則を単一ソースで満たし、総ページ数がライブの `.slide` 数と一致するか。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、頂点(結論)ノード・第2階層(MECE な根拠)・任意の末端(裏付け)を、親子コネクタで結んだ**縦(トップダウン)ツリー**をコアの新レイアウト(`data-layout="tree"`)として提供しなければならない(MUST)。
- **FR-002**: システムは、根を左端・枝を右へ展開する**横(左→右)ツリー**を付録スライドとして提供しなければならない(MUST)。
- **FR-003**: 縦・横のツリーは、**単一の再利用可能なコンポーネント**(共通のノード/コネクタ表現)を向き替えで用いなければならない(MUST)。
- **FR-004**: ノードとコネクタの配色・余白・角丸・文字サイズ・境界は、すべて DS のトークン(役割変数)由来でなければならず、生のカラーコード(hex/rgb/hsl)や余白系プロパティの生 px を含んではならない(MUST NOT)。
- **FR-005**: 両ツリーのすべてのノード・コネクタ・テキストは、1280×720 のスライド枠内に見切れ・はみ出し・重なりなく収まらなければならない(MUST)。
- **FR-006**: ツリーの各ノードのサンプルコピーは、ピラミッド・プリンシプルの構造(頂点=支配的メッセージ、第2階層=MECE な根拠、末端=個別の裏付け)を体現しなければならない(MUST)。
- **FR-007**: 印刷(PDF)出力は、既存の `@media print`(1スライド=1ページ)の単一ソースを用い、新レイアウト用の新規ページ CSS を追加してはならない(MUST NOT)。総 PDF ページ数はライブの `.slide` 数と一致しなければならない(MUST)。
- **FR-008**: コアレイアウトの追加に伴い、`slides/*.html` は生成器で再生成されなければならず(手編集禁止)、生成物とライブデッキが一致しなければならない(MUST)。
- **FR-009**: デッキのスライド総数と可視のカウント表示(該当があれば)は、新スライド追加後の値に整合していなければならない(MUST)。
- **FR-010**: 既存の検証スイート(トークン Lint、相互参照、カバレッジ、スライド数、視覚回帰、アクセシビリティ、印刷、PDF 生成/ページ数)がすべて緑でなければならない(MUST)。
- **FR-011**: 文字色と背景色のコントラストは WCAG AA を満たさなければならない(MUST)。
- **FR-012**: 新規のランタイム依存を追加してはならない(MUST NOT)。

### Key Entities

- **ツリー(Tree)**: 階層構造全体。1つの根、複数の枝からなる。向き(縦/横)の属性を持つ。
- **ノード(Node)**: ツリーの要素。役割(root=結論 / branch=根拠 / leaf=裏付け)と本文を持つ。
- **コネクタ(Connector)**: 親子ノードを結ぶ視覚要素。方向(縦=上下 / 横=左右)を持つ。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 縦・横いずれのツリーも、1280×720 スライド枠内に 100% 収まり(見切れ・はみ出し 0 件)、視覚回帰スナップショットが基準内で一致する。
- **SC-002**: 読み手がツリー1枚から「結論 → 根拠グループ → 裏付け」の親子階層を、追加説明なしで識別できる(頂点1・第2階層複数・親子コネクタが明示される)。
- **SC-003**: PDF の総ページ数がライブの `.slide` 総数(27)と一致し、`npm run verify` が全工程緑になる。
- **SC-004**: すべての色・余白がトークン由来で、トークン Lint 違反 0 件。文字コントラストが AA を満たす。
- **SC-005**: 第2階層のノード数(例: 2〜5)や末端の有無を差し替えても、レイアウトが枠内に収まり破綻しない。

## Assumptions

- ツリーの内容モデルはピラミッド・プリンシプル(結論先出し・MECE)とする(ユーザー指定)。用途に応じ利用者が文言を差し替える前提。
- 縦(トップダウン)をコアの新レイアウト、横(左→右)を付録スライドとする(ユーザー選択「コア新レイアウト＋付録」)。
- 既定のサンプルは 3階層程度(頂点1・第2階層3・任意の末端)を基準とし、深さ・枝数は再利用時に調整可能とする。
- 既存の 1280×720 キャンバス、12カラム×3行グリッド、DS トークン、`@media print` 単一ソース、生成器による `slides/*.html` 生成の枠組みを踏襲する。
- ツリーは通常(明色)スライド前提。濃色系スライドの反転規則の対象外とする。
- コアレイアウト追加後のデッキ総数は 27 スライド(現行 25 + 縦コア1 + 横付録1)を基準とする。
