# Feature Specification: HTML→PDF 生成物を Claude Code が確認・評価するフロー

**Feature Branch**: `008-html-pdf-preview`

**Created**: 2026-07-13

**Status**: Draft

**Input**: User description: "htmlをpdf化して確認するフローを追加。ブラウザで確認するのもできれば。PDF化されたものをClaude Codeが読み込み、生成内容が正しいか、適切かを評価する。ブラウザで表示して確認するのも可能であればやらせたい。人が確認するのは副次的で、Claude Codeが確認することに主眼を置く。"

## User Scenarios & Testing *(mandatory)*

このデックの正本は `index.html`（16レイアウト＋付録）です。レイアウトやコンポーネントを変更したとき、その結果が実出力（PDF）で崩れていないか、デックの設計意図に沿っているかを、**Claude Code 自身が**確認・評価できるようにしたい。従来は人が目視するしかなかったが、本機能では正本デックを PDF 化し、Claude Code がそのページ画像を読み込んで「描画が正しいか」「設計意図に適合しているか」を評価し、所見をレポートする。可能ならブラウザ表示でも同様に確認する。人による最終確認は副次的な位置づけとし、確認の主体は Claude Code に置く。

評価の対象範囲は次の 2 点とする（内容そのものの良し悪し＝文言・メッセージの妥当性は範囲外）:

- **描画破綻**: 見切れ（オーバーフロー）、1枚が複数ページに分裂、共通フレーム（取扱区分・著作権・ページ番号）の欠落、要素の重なり・文字化け・レイアウト崩れ。
- **設計意図適合**: デックが掲げる設計意図（DS 準拠の配色・トークン、アクションタイトル、視覚階層、1スライド1メッセージ、可読性・コントラスト）への視覚レベルでの適合。

### User Story 1 - Claude Code が PDF を読み込み描画・設計適合を評価する (Priority: P1)

Claude Code が確認フローを起動すると、正本デックから PDF が生成され、Claude Code はその各ページ画像を読み込む。Claude Code は各スライドについて「描画破綻の有無」と「設計意図への適合」を評価し、スライド単位の所見（合否と、問題があればその内容・該当スライド）を構造化してレポートする。

**Why this priority**: 本機能の主眼は「Claude Code による確認」。実出力での崩れ・設計逸脱を人手を介さず Claude Code が捉えられることが中核価値であり、これ単体で成立する。

**Independent Test**: 正本デックに対し確認フローを1回起動し、生成 PDF の全ページを Claude Code が読み込み、全スライド分のスライド単位所見（描画破綻・設計適合の判定）を含むレポートが得られれば、この価値は単独で成立する。

**Acceptance Scenarios**:

1. **Given** 正本デック `index.html` が有効な状態、**When** Claude Code が確認フローを起動、**Then** PDF が生成され、Claude Code が全ページ画像を読み込める形で提供される。
2. **Given** 読み込んだ PDF ページ画像、**When** Claude Code が評価、**Then** スライド単位で「描画破綻の有無」「設計意図適合の可否」を判定した構造化レポート（該当スライド・所見付き）が出力される。
3. **Given** あるスライドに見切れ・ページ分裂・フレーム欠落が生じている、**When** Claude Code が評価、**Then** 当該スライドを描画破綻として指摘し、位置・症状を所見に記す。
4. **Given** 設計意図から逸脱した表示（例: コントラスト不足、アクセント多用、視覚階層の崩れ）がある、**When** Claude Code が評価、**Then** 当該スライドを設計不適合として指摘する。
5. **Given** すべてのスライドが正常、**When** Claude Code が評価、**Then** レポートは全スライド合格を示し、指摘 0 件となる。

---

### User Story 2 - Claude Code がブラウザ表示で確認する (Priority: P2)

Claude Code が確認フローの一環でデックをローカルにブラウザ表示し、画面表示（および発表モード・注釈トグルなどの対話挙動）を観察して、PDF では捉えにくい表示・挙動を評価に加える。

**Why this priority**: 対話挙動やスクリーン専用表示は PDF に現れない。ブラウザ確認で評価の死角を補完できるが、配布実体は PDF のため P1 に次ぐ。ユーザー要望では「可能であれば」と条件付き。

**Independent Test**: Claude Code がデックをブラウザ表示し、少なくとも1つのスライドと1つの対話挙動（発表モード遷移など）を観察して所見に反映できれば単独で成立する。

**Acceptance Scenarios**:

1. **Given** 作業ツリー、**When** Claude Code がローカル表示を起動しブラウザで開く、**Then** デック全体を画面表示で観察できる。
2. **Given** ブラウザ表示中、**When** Claude Code が発表モード・注釈トグルを操作、**Then** それぞれの挙動を観察し、描画・設計適合の所見に反映できる。

---

### User Story 3 - 確認フローを繰り返し実行でき既知の不備を検出できる (Priority: P3)

同じ確認フローを繰り返し起動でき、既知の不備（意図的に注入した見切れ・ページ分裂など）に対して Claude Code の評価が安定して「不備あり」を返す。決定的な下地チェック（PDF が生成でき、ページ数がスライド枚数に一致）は自動判定として `verify` に組み込める。

**Why this priority**: 主観的評価（Claude Code の判断）を、決定的な下地チェックと既知不備の検出可能性で裏打ちし、フローの信頼性を担保する。P1/P2 の確認能力の上積み。

**Independent Test**: 既知の不備を注入した版に対し確認フローを起動し、Claude Code が当該不備を検出できること、および正常版で下地チェックが PASS することを確認できれば単独で成立する。

**Acceptance Scenarios**:

1. **Given** 正常なデック、**When** 下地チェックを実行、**Then** PDF 生成が成功し、ページ数がスライド枚数に一致して PASS する。
2. **Given** 見切れ・ページ分裂を注入した版、**When** Claude Code が確認フローで評価、**Then** 当該不備を「描画破綻」として検出・指摘する。

---

### Edge Cases

- スライド枚数が増減しても、確認フローは固定枚数を前提とせず実枚数に追随し、ページ画像の読み込み・照合ができること。
- ページ画像が読み込めない／解像度が低く判定できない場合、Claude Code は「評価不能」を明示し、誤って合格としないこと。
- 画面表示専用の余白・影・ツールバー・注釈レイヤーが PDF に混入して版面を崩さないこと（混入時は描画破綻として検出対象）。
- ブラウザ確認が環境的に不可能な場合でも、PDF による確認（P1）は成立し、フロー全体が失敗しないこと。
- Claude Code の評価は主観判断を含むため、指摘は該当スライドと具体的症状を伴い、追跡・再現可能な形で記録されること。
- 生成 PDF（`dist/deck.pdf`）は追跡対象とし、per-slide PNG 等その他の一時生成物はリポジトリを汚さないよう追跡対象外とすること。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、正本デック `index.html` から PDF を生成する、単一の起動可能な確認フローを提供しなければならない。
- **FR-002**: 確認フローは、生成 PDF を Claude Code が各ページ画像として読み込める形で提供しなければならない。
- **FR-003**: Claude Code は、PDF の全ページを読み込み、スライド単位で「描画破綻の有無」と「設計意図への適合」を評価しなければならない。
- **FR-004**: 評価結果は、スライド単位の合否・該当箇所・所見を含む構造化されたレポートとして出力されなければならない。
- **FR-005**: 描画破綻の評価は、少なくとも 見切れ／複数ページ分裂／共通フレーム欠落／要素の重なり・レイアウト崩れ を対象に含めなければならない。
- **FR-006**: 設計意図適合の評価は、デックが掲げる設計意図（DS 準拠の配色・トークン、アクションタイトル、視覚階層、1スライド1メッセージ、可読性・コントラスト）への視覚レベルの適合を対象に含めなければならない。
- **FR-007**: 確認フローは、可能な環境では Claude Code によるブラウザ表示での確認（対話挙動を含む）を行えなければならない。ブラウザ確認が不能でも PDF による確認は成立しなければならない。
- **FR-008**: 確認フローは、その時点のスライド枚数に自動追随し、枚数のハードコードに依存してはならない。
- **FR-009**: 確認フローには、決定的な下地チェック（PDF が例外なく生成でき、ページ数がスライド枚数に一致）を含め、既存の検証手順（`verify`）に組み込み可能でなければならない。
- **FR-010**: 確認フローは、既存の印刷用挙動（`@media print` による 1枚=1ページ）を唯一の版面定義として再利用し、版面を二重定義してはならない。
- **FR-011**: 確認フローの生成 PDF（`dist/deck.pdf`）は、既存の PPTX 成果物（`dist/sample-deck.pptx`）と同様にバージョン管理の追跡対象とし、ビルド不要でレビュー・閲覧できるようにしなければならない。その他の一時生成物（per-slide PNG 等）は追跡対象外とする。
- **FR-012**: 確認フローの起動方法・出力（レポート）の見方・評価観点は、リポジトリ内の作成者向けドキュメント（README 等）に記載されなければならない。
- **FR-013**: ページ画像が判定に不十分な場合、Claude Code は当該スライドを「評価不能」と明示し、合格として扱ってはならない。
- **FR-014**: 人による最終確認は副次的であり、確認フローは人の介在を必須とせず Claude Code 主体で完結できなければならない。

### Key Entities *(include if data involved)*

- **正本デック**: `index.html`。確認対象の単一情報源。スライド群と共通フレームを含む。
- **PDF 出力**: 確認フローが生成する配布相当の成果物。ページ＝スライドに対応。Claude Code が読み込む評価入力。`dist/deck.pdf` として追跡対象。
- **ページ画像**: Claude Code が視覚評価するために読み込む各スライドの画像表現。
- **評価レポート**: Claude Code が出力する、スライド単位の合否・該当箇所・所見を含む構造化された確認結果。
- **確認手順ドキュメント**: 起動方法・レポートの見方・評価観点を記した作成者向け記述（README 等）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 確認フロー1回の起動で、生成 PDF の全ページ（=全スライド）を Claude Code が読み込み、100% のスライドに対して所見を出力できる。
- **SC-002**: 生成 PDF のページ数が、デックの実スライド枚数と 100% 一致する。
- **SC-003**: 意図的に注入した描画破綻（見切れ・ページ分裂・フレーム欠落）を、Claude Code の評価が 100% 検出し、該当スライドを特定できる。
- **SC-004**: すべてのスライドが正常な版では、Claude Code の描画破綻に関する誤検出（False Positive）が 0 件である。
- **SC-005**: 評価レポートは、各スライドについて「描画破綻の有無」と「設計適合の可否」を、該当箇所・所見付きで機械可読・人可読の双方に扱える構造で示す。
- **SC-006**: ブラウザ確認が可能な環境では、Claude Code が少なくとも全スライドの画面表示と1つ以上の対話挙動を観察して所見に反映できる。ブラウザ確認が不能でも PDF 確認は完了する。
- **SC-007**: 版面（1枚=1ページ）の定義がリポジトリ内で二重化されず、単一の定義から PDF が生成される。
- **SC-008**: 確認フローの起動から評価レポート取得までを、追加の口頭説明なしにドキュメント記載の手順のみで実施できる。

## Assumptions

- 確認の主体は Claude Code（本リポジトリで動作する AI エージェント）であり、人による確認は任意・副次的である（confidence: high）。
- 「評価」の範囲は 描画破綻＋設計意図適合 の 2 点に限定し、文言・メッセージ内容そのものの良し悪しは範囲外とする（confidence: high、ユーザー確認済み）。
- 設計意図の基準は、デックが既に掲げる設計原則（`docs/practices.md`／注釈・DS トークン等）に依拠し、新たな評価基準体系は策定しない（confidence: medium）。
- 版面の正は既存の `@media print`（1スライド=1ページ）であり、PDF 化はこれを再利用する（confidence: high）。
- Claude Code の視覚評価は PDF/ブラウザのページ画像を入力とし、既存の PDF スモークテスト（Playwright による PDF 生成）を出発点にできる（confidence: medium）。
- ブラウザ確認は既存のショーケース表示（発表モード・注釈トグルを含む）の観察を指し、新規ビューアの実装は含まない。実行環境が無い場合はスキップ可能（confidence: medium）。
- 生成 PDF（`dist/deck.pdf`）は既存 PPTX 成果物と同様に追跡し、per-slide PNG 等その他の一時生成物は `.gitignore` で管理外とする（confidence: high、ユーザー確認済み）。
- Claude Code の評価は主観判断を含み完全な決定性は持たないため、信頼性は決定的な下地チェック（枚数一致・生成成功）と既知不備の検出可能性で裏打ちする（confidence: high）。
- PPTX 出力（別途 `scripts/pptx/`）とは独立した確認フローであり、本機能では PPTX を扱わない（confidence: high）。
