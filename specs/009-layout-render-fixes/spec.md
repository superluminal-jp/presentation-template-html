# Feature Specification: レイアウト描画の修正と構図の統一

**Feature Branch**: `009-layout-render-fixes`

**Created**: 2026-07-13

**Status**: Draft

**Input**: User description: "改善案を適用"

## Context

ショーケースデック(`index.html`)を PDF レンダリングで確認した結果、(1) 仕様どおりに描画されない2件の不具合、(2) スライド間で本文の縦位置が揃わず余白が間延びする構図の癖、が見つかった。本フィーチャーはこれらを是正し、配布物としての正確性と一貫性を高める。既存の制約(DS トークン準拠・ハードコード0・WCAG 2.2 AA・ネイティブ寸法の視覚回帰・印刷/PDF は1スライド=1ページ・PDF ページ数=ライブの `.slide` 数)は不変とする。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 描画不具合の是正(凡例色・濃色地の可読性) (Priority: P1)

閲覧者がデックを PDF または画面で見たとき、凡例の色分けが実際のデータ系列と一致し、どのスライドでも取扱区分などのフレーム表示が判読できる。

**Why this priority**: 凡例色の不一致はデータの誤読を招き、濃色地でのフレーム不可読はアクセシビリティ(コントラスト)と配布物要件(取扱区分の明示)に反する。いずれも「正しく見える」ことが前提の欠陥で、最優先。

**Independent Test**: 付録D(多系列チャート)の凡例と全面画像スライドを単独で確認すれば、系列色の一致とフレーム文字の判読性を検証できる。

**Acceptance Scenarios**:

1. **Given** 付録Dの多系列チャート, **When** スライドを描画する, **Then** 凡例の各スウォッチ色が対応する系列(企画=青/開発=緑/運用=橙)と一致する。
2. **Given** 濃色地の全面画像スライド, **When** スライドを描画する, **Then** 右上の取扱区分(組織限定)を含むフレーム文字が背景に対し WCAG 2.2 AA を満たすコントラストで判読できる。
3. **Given** 系列色を用いる他の付録スライド(ダッシュボード・グラフ事例集), **When** スライドを描画する, **Then** 凡例・系列・データ点の色対応が全て一致している。

---

### User Story 2 - 構図の統一(本文の縦位置とビッグナンバーの階層) (Priority: P2)

発表者がスライドを次々に送ったとき、本文が見出しに続けて一定の位置から始まり、視線の落ち着き先がスライド間でぶれない。単一指標のスライドでは数値が主役として際立つ。

**Why this priority**: 本文が縦中央に沈み見出し直下に大きな空白が空く癖が複数レイアウトで生じ、スライド間の一貫性と情報密度を損ねている。ビッグナンバーは訴求の中心が弱い。正確性の問題ではないため P2。

**Independent Test**: ビッグナンバー・まとめ・プロセス・タイムライン・図表・付録Fを並べ、本文開始位置の揃いとビッグナンバーの視覚的優位を確認できる。

**Acceptance Scenarios**:

1. **Given** 本文量が少ないコンテンツ系スライド, **When** スライドを描画する, **Then** 本文は見出し直下から始まり、見出しと本文の間に間延びした空白が生じない。
2. **Given** 縦中央寄せが意図されたレイアウト(章扉・引用・ビッグナンバー), **When** スライドを描画する, **Then** それらは従来どおり意図した中央寄り構図を保つ。
3. **Given** ビッグナンバーのスライド, **When** スライドを描画する, **Then** 主指標の数値が同一スライド内で最も大きい視覚要素として際立ち、補足(内訳)と1つのまとまりとして近接配置される。

---

### User Story 3 - 仕上げの微調整(疎なスライドと折り返し) (Priority: P3)

閲覧者がデック全体を通覧したとき、極端に空白の目立つスライドや不自然な行折り返しがなく、通しての体裁が均質に感じられる。

**Why this priority**: 体裁上の細部で、機能・正確性には影響しない。任意対応。

**Independent Test**: 表紙・クロージング・付録F・付録H を確認し、構図の均質さと折り返しの自然さを検証できる。

**Acceptance Scenarios**:

1. **Given** 下半分が大きく空く付録スライド(F・H), **When** スライドを描画する, **Then** 空白が意図的な余白として成立するか、他の付録と密度が揃う。
2. **Given** クロージングの CTA, **When** スライドを描画する, **Then** 語の途中で不自然に折り返さない。
3. **Given** 表紙, **When** スライドを描画する, **Then** 主要素が上下端に貼り付かず、意図した構図として配置される。

---

### Edge Cases

- 縦位置の既定を「上詰め」に変えたとき、本文量が多く行3(フッタ)まで詰まっているスライド(出典・ダッシュボード等)で見切れ(オーバーフロー)を起こさないこと。
- ビッグナンバーの数値拡大時、単位・内訳と重ならず安全余白の内側に収まること。
- 凡例色の是正が、系列色を使わない既存の凡例(当月/前月/目標などの中立スウォッチ)に影響しないこと。
- 全変更後も PDF のページ数がライブの `.slide` 総数と一致し続けること。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 多系列チャートの凡例スウォッチは、対応するデータ系列と同一の色で描画されなければならない(青/緑/橙の系列色対応)。
- **FR-002**: 系列色を用いる全スライド(ダッシュボード・グラフ事例集を含む)で、凡例・系列・データ点の色対応が一貫していなければならない。
- **FR-003**: 濃色地のスライド(全面画像)で、右上の取扱区分を含むフレーム文字が背景に対し WCAG 2.2 AA を満たすコントラストで判読可能でなければならない。
- **FR-004**: 本文量が少ないコンテンツ系スライドで、本文は見出しに続けて配置され、見出し直下に間延びした空白を生じさせてはならない。
- **FR-005**: 縦中央寄せが設計意図であるレイアウト(章扉・引用・ビッグナンバー)は、その意図した構図を保持しなければならない。
- **FR-006**: ビッグナンバーのスライドで、主指標の数値が同一スライド内の最大の視覚要素となり、補足情報と近接した1つのまとまりとして配置されなければならない。
- **FR-007**: 下半分が大きく空く付録スライドは、意図的な余白として成立させるか、他の付録スライドと密度を揃えなければならない(P3)。
- **FR-008**: クロージングの CTA は語の途中で不自然に折り返してはならない(P3)。
- **FR-009**: 全ての変更はデジタル庁 DS トークンを参照して行い、生の px 等のハードコード値を新たに導入してはならない(トークン lint を通過)。
- **FR-010**: 全ての変更後も、ネイティブ寸法の視覚回帰・アクセシビリティ・印刷テストが通過し、PDF のページ数がライブの `.slide` 総数と一致しなければならない。
- **FR-011**: 印刷/PDF の「1スライド=1ページ」規則および `@media print` の単一ソースは変更してはならない。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 系列色を用いる全チャートで、凡例スウォッチと系列の色不一致が 0 件。
- **SC-002**: 濃色地スライドのフレーム文字のコントラスト比が 4.5:1 以上(WCAG 2.2 AA、通常テキスト)。
- **SC-003**: 本文の縦位置の癖が是正され、対象スライド(ビッグナンバー・まとめ・プロセス・タイムライン・図表・付録F)で見出し直下から本文が始まる割合が 100%。
- **SC-004**: ビッグナンバーの主指標が、同一スライド内で最大のフォントサイズ要素であることをレビューで確認できる。
- **SC-005**: `npm run verify`(トークン lint・視覚・a11y・印刷・PDF ページ数検証を含む)が全て通過。
- **SC-006**: 変更前後で PDF の総ページ数が不変(=ライブの `.slide` 総数)。

## Assumptions

- 「改善案を適用」とは、直前のレビューで提示した提案一式を指す。P1(描画不具合)と P2(構図統一)を必須、P3(仕上げ)を任意対応とする(提案の優先度構成に準拠)。
- ビッグナンバーの数値拡大は、DS 型スケールの最大トークンをそのまま用いるか、トークン由来の派生値(トークンを基点とした計算値)で表現し、生 px は用いない。派生値がトークン lint に抵触する場合は、構図の締め(余白圧縮・近接配置)で視覚的優位を確保する代替に切り替える。
- 縦位置の既定変更は本文帯(行2)の配置指定に対して行い、行1(見出し)・行3(フッタ)のグリッド構造は変更しない。
- スライドの追加・削除は行わない(既存25スライドの体裁調整のみ)。PPTX 出力側の同等修正は本フィーチャーの範囲外とし、必要なら別途対応する。
- 検証は既存の `npm run verify` パイプライン(Playwright + axe-core + トークン lint + PDF 検証)を正とする。
