# Feature Specification: スライドテンプレート → PowerPoint テンプレート(.potx)変換スクリプト

**Feature Branch**: `005-pptx-export-script`

**Created**: 2026-07-05

**Status**: Draft

**Input**: User description: "スレイドテンプレートを pptx に変換するスクリプトを作成"(2026-07-05 追記: 出力形式は pptx ではなく potx とする)

## Clarifications

### Session 2026-07-05

- Q: HTML のダミー例文(「見出し例」等)を `.potx` のプレースホルダーにどう反映するか? → A: PowerPoint 標準のプレースホルダー・プロンプトテキスト(「クリックしてタイトルを入力」のような、入力開始で消え印刷/出力に含まれないヒント)として扱う。実テキストの事前入力はしない。
- Q: DS トークンの配色・フォントを PowerPoint のどの階層に反映するか? → A: PowerPoint のテーマ配色・フォント定義(theme1.xml)自体を DS トークンで上書きし、プレースホルダー外で利用者が新規に描画する図形・テキストにも既定でDS トークン由来の色/フォントが適用されるようにする。
- Q: `.potx` の互換性検証対象はどのアプリケーションか? → A: Microsoft PowerPoint デスクトップ版(現行および 1 つ前のメジャーバージョン、Windows/Mac)のみを検証対象とする。LibreOffice Impress や Google スライド等での動作は保証しない。
- Q: 生成される `.potx` にアクセシビリティ要件を課すか? → A: 課す。画像プレースホルダーには編集可能な代替テキスト用のヒントを設定し、図形の読み上げ順序はタイトル→本文→補足コンテンツの順に従うこと。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - HTML の 16 レイアウトを PowerPoint の「スライドレイアウト」として使えるようにする (Priority: P1)

`slides/` 配下にある 16 種類の HTML レイアウトテンプレート(表紙・目次・章扉・本文…等、まだ具体的な発表内容が入っていない雛形)を、PowerPoint テンプレートファイル(.potx)に変換したい。配布された `.potx` を PowerPoint で開き「新しいスライド」から選ぶと、16 レイアウトがそれぞれ選択肢として現れ、各レイアウトのプレースホルダー(タイトル・本文・画像枠等)にその場でテキストや画像を流し込んで新しいプレゼンテーションを作成できる必要がある。

**Why this priority**: `.potx` 化は「HTML を触れない利用者が PowerPoint だけで本テンプレートを使い続けられる」ための配布経路であり、本フェーズの中核要求。

**Independent Test**: 変換スクリプトを実行して生成された `.potx` を PowerPoint で開き、スライドマスター表示で 16 個のカスタムレイアウトが存在することを確認する。次に新規プレゼンテーションをこのテンプレートから作成し、「ホーム > 新しいスライド」のレイアウト一覧に 16 レイアウトが表示され、各レイアウトを適用したスライドでタイトル・本文プレースホルダーにテキスト入力できることを確認する。

**Acceptance Scenarios**:

1. **Given** `slides/` 配下の 16 レイアウト HTML ファイル、**When** 変換スクリプトを実行する、**Then** 1 つの `.potx` ファイルが生成され、その中に 16 個のスライドレイアウトが(HTML ファイルと同じ命名・順序に対応する形で)含まれる。
2. **Given** 生成された `.potx`、**When** PowerPoint で開き「新しいスライド」を挿入する、**Then** 16 レイアウトすべてがレイアウト選択ギャラリーに表示され、選択すると対応する構造(タイトル位置・本文位置・区画分割等)のスライドが挿入される。
3. **Given** いずれかのレイアウトを適用したスライド、**When** タイトルや本文プレースホルダーをクリックして文字を入力する、**Then** 画像化されたテキストではなく、通常の PowerPoint プレースホルダーとして編集できる。

---

### User Story 2 - DS トークン由来の配色・書体をスライドマスター/レイアウトに反映する (Priority: P2)

`.potx` 内の各レイアウトの背景色・見出し色・本文色・フォントが、元の HTML テンプレートが参照しているデジタル庁デザインシステム(DS)トークンの実測値と一致していることを確認したい。

**Why this priority**: 配色・書体が崩れると、PowerPoint 側で新規作成したスライドが元テンプレートとブランド不整合になり、`.potx` を配布する意味が薄れる。

**Independent Test**: 特定のレイアウト(例: 表紙・本文・比較)について、HTML の計算済みスタイル(DS トークン値)と `.potx` 内の対応するレイアウトのテーマカラー・フォント設定を比較し、一致することを確認する。

**Acceptance Scenarios**:

1. **Given** DS トークンで定義されたテーマカラー(キーカラー、本文文字色、背景色等)、**When** 変換する、**Then** `.potx` の PowerPoint テーマ配色定義(theme1.xml)の対応するスロットが DS トークンの RGB 値と一致する。
2. **Given** Noto Sans JP を用いた見出し・本文、**When** 変換する、**Then** `.potx` のフォント指定が Noto Sans JP(または明示的なフォールバック)になっている。
3. **Given** 16 レイアウトそれぞれ、**When** 一括変換する、**Then** レイアウトごとの構造的な違い(表紙/章扉/本文/比較/図表/まとめ等)がプレースホルダーの配置・種類として区別できる。

---

### User Story 3 - 変換できない要素を明示的に報告する (Priority: P3)

スクリプトがレイアウトを完全に再現できない箇所(複雑なダッシュボードグラフ、SVG 装飾等)がある場合、無言で崩れた出力を返すのではなく、変換ログで利用者に知らせてほしい。

**Why this priority**: 変換漏れに気づかず `.potx` を配布してしまうリスクを防ぐ安全網であり、P1/P2 が成立すれば単独でも価値を持つ補助機能。

**Independent Test**: 意図的に複雑な図表(例: `10-dashboard.html` のグラフ)を含むレイアウトを変換し、標準出力またはログファイルに当該レイアウト名と未対応要素の種類が記録されることを確認する。

**Acceptance Scenarios**:

1. **Given** ラスター画像化にフォールバックした要素、**When** 変換が完了する、**Then** どのレイアウトのどの要素が画像化されたかがログに出力される。
2. **Given** 変換中にスタイル値(DS トークン)を解決できない要素、**When** 変換が完了する、**Then** エラーで停止せず、警告としてログに残し処理を継続する。

---

### Edge Cases

- HTML レイアウトに繰り返し要素(比較表の列数、タイムラインの項目数等、レイアウトによって数が可変な部分)がある場合、`.potx` のレイアウトはその「型」を代表する 1 パターンとして固定し、可変数への追従は範囲外とする。
- レイアウト内に外部画像参照が壊れている(ファイルが存在しない)場合、画像プレースホルダー(枠のみ)を挿入し警告を出す。
- 入力 HTML が 16 レイアウトの命名・クラス規約に従っていない(未知のレイアウトクラス)場合、汎用レイアウトとして最善努力で変換し警告を出す。
- HTML 側の注釈フレーム要素(分類ラベル・著作権表示・ページ番号)は、スライドマスター側の共通要素として全レイアウトに引き継がれる。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは `slides/` 配下の 16 レイアウト HTML ファイル(`01-title.html`〜`16-closing.html`)を入力として受け取り、コマンド実行(CLI スクリプト)で `.potx` 変換を開始できなければならない。
- **FR-002**: システムは各 HTML レイアウトを 1 対 1 で、出力 `.potx` 内の 1 つのカスタムスライドレイアウトに変換しなければならない。
- **FR-003**: システムは出力される `.potx` のスライドサイズを 13.333インチ×7.5インチ(16:9、docs/requirements.md の非機能要件と整合)に設定しなければならない。
- **FR-004**: システムは各レイアウトのタイトル・本文・画像領域等を、PowerPoint のプレースホルダー(またはそれに準ずる編集可能な図形)として配置しなければならず、画像として焼き込んではならない(FR-008 のフォールバック対象を除く)。プレースホルダーには実テキストを事前入力せず、PowerPoint 標準のプロンプトテキスト(入力開始で消え、印刷・出力に含まれないヒント)のみを設定しなければならない。
- **FR-005**: システムは `styles/tokens.semantic.css` 等で定義された DS トークンの実効値(色・フォントファミリー・フォントサイズ・行間)を解決し、`.potx` の PowerPoint テーマ配色・フォント定義(色/フォントスキーム)自体を DS トークン値で上書きしなければならない。これにより、プレースホルダー外で利用者が新規に描画する図形・テキストにも既定で DS トークン由来の色/フォントが適用される。
- **FR-006**: システムはハードコードされた色・フォント値を用いず、HTML 側が参照している CSS 変数(トークン)の実測値のみを変換元として使用しなければならない。
- **FR-007**: システムは 16 レイアウトすべて(表紙/目次/章扉/本文/比較/図表/まとめ/リファレンス/ビッグナンバー/ダッシュボード/タイムライン/マトリクス/プロセス/引用/全面画像/クロージング)に対して変換ロジックを持たなければならない。
- **FR-008**: システムはネイティブな図形・表・プレースホルダーとして再現できない要素(複雑なグラフ、SVG 装飾等)についてはラスター画像へのフォールバック変換を行い、その旨をログに記録しなければならない。
- **FR-009**: システムは変換中に発生したエラー・警告(未対応レイアウト、画像欠落、トークン解決失敗等)を、処理を中断せずにログとして出力しなければならない。
- **FR-010**: システムは変換処理をローカル環境のみで完結させ、外部ネットワークアクセスに依存してはならない(既存の非機能要件 Performance と整合)。
- **FR-011**: システムは出力ファイル名・出力先ディレクトリを利用者が指定できなければならない(未指定時は既定の出力先を用いる)。
- **FR-012**: システムは各レイアウトに、元の HTML と対応する分かりやすい名前(例:「表紙」「本文」「比較」等)を設定し、PowerPoint のレイアウト選択ギャラリーで名前から用途を判別できなければならない。
- **FR-013**: システムは分類ラベル・著作権表示・ページ番号等の共通フレーム要素を、個々のレイアウトではなくスライドマスター側に一度だけ定義し、全レイアウトに継承させなければならない。
- **FR-014**: システムは画像・図表プレースホルダーに、利用者が編集できる代替テキスト(alt text)のヒントを設定しなければならない。
- **FR-015**: システムは各レイアウト内の図形の読み上げ順序(タブ順/アクセシビリティ順序)を、タイトル→本文→補足コンテンツの順になるよう設定しなければならない。

### Key Entities

- **スライドテンプレート(HTML)**: `slides/` 配下の 16 レイアウトファイル。DS トークン(CSS 変数)を参照した見出し・本文・図表・フレーム要素(分類ラベル・著作権表示・ページ番号)を持つ。
- **PowerPoint テンプレート(.potx)**: 変換後の出力物。1 つのスライドマスターと、HTML の 16 レイアウトに対応する 16 個のカスタムスライドレイアウトから構成される。実際の発表内容(スライドインスタンス)は含まない。
- **レイアウト対応表**: HTML の各レイアウトクラス(例: `.layout-title`, `.layout-compare`)と `.potx` 上のプレースホルダー配置ルール(種類・位置・サイズ・書式)を対応付けるマッピング定義。
- **変換ログ**: 変換実行ごとに生成される、成功・警告・フォールバック内容を記録した実行記録。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 16 レイアウトすべてについて、変換スクリプト実行後に PowerPoint で開ける `.potx` が生成され、スライドマスター表示で 16 個のカスタムレイアウトが確認できる(変換成功率 100%、クラッシュ・例外終了なし)。
- **SC-002**: PowerPoint の「新しいスライド」レイアウト一覧に 16 レイアウトすべてが表示され、名前からどのレイアウトか判別できる。
- **SC-003**: 生成されたレイアウトのタイトル・本文領域の 95% 以上が、画像化されず選択・編集可能なプレースホルダーとして存在する。
- **SC-004**: 各レイアウトの配色(背景色・見出し色・本文色)が元 HTML の DS トークン実測値と RGB 値で完全一致する。
- **SC-005**: 変換にかかる時間は、16 レイアウト一括変換で 30 秒以内に完了する。
- **SC-006**: 変換できなかった要素がある場合、利用者はログを見るだけで「どのレイアウトの何が完全再現されなかったか」を追加調査なしに把握できる。
- **SC-007**: 生成された `.potx` は Microsoft PowerPoint デスクトップ版(現行および 1 つ前のメジャーバージョン、Windows/Mac)で開いたとき、スライドマスター・16 レイアウト・プレースホルダー・テーマ配色がすべて意図通りに表示される。
- **SC-008**: 画像・図表を含む全レイアウトにおいて、画像プレースホルダーに代替テキストのヒントが設定されており、読み上げ順序がタイトル→本文→補足コンテンツの順になっている。

## Assumptions

- 変換対象は `slides/` にある 16 レイアウトの HTML テンプレートそのものであり、具体的な発表内容が入った完成プレゼンテーションではない。出力もスライドインスタンスの集合ではなく、PowerPoint の「新規作成」時に選べるレイアウト定義一式(`.potx`)である。
- 各 HTML レイアウトが持つプレースホルダー的なダミーテキスト(見出し例・本文例等)は、`.potx` 側では実テキストとして事前入力せず、PowerPoint 標準のプレースホルダー・プロンプトテキスト(入力開始で消え、印刷・出力に含まれないヒント)として扱う(2026-07-05 Clarifications 参照)。
- 忠実度の基準は「レイアウト構造・プレースホルダー種別・DS トークン由来の配色/書体が編集可能な形で再現されること」であり、ピクセル単位の完全一致(装飾的な影・グラデーション等)までは求めない。複雑な装飾やダッシュボードグラフはラスター画像へのフォールバックを許容する(FR-008)。
- 実行環境には Python と `python-pptx` ライブラリ(docs/requirements.md で第一候補として言及)がインストールされている、またはスクリプトの依存関係としてインストール可能であることを前提とする。`python-pptx` は `.potx` の生成もサポートすることを前提とする(`.pptx` と同じ OOXML 構造で拡張子とコンテンツタイプのみ異なる)。
- 入力 HTML は本リポジトリの既存レイアウト規約(`slides/01-title.html` 〜 `slides/16-closing.html` のクラス命名・構造)に準拠していることを前提とし、全く異なる HTML 構造への汎用対応は範囲外とする。
- レイアウト内の繰り返し要素(比較表の列数等、可変長のコンテンツ)は、代表的な 1 パターンのみを `.potx` のレイアウトとして固定する。可変数バリエーションの自動生成は範囲外とする。
- 動作検証対象は Microsoft PowerPoint デスクトップ版(現行および 1 つ前のメジャーバージョン、Windows/Mac)のみとする。LibreOffice Impress や Google スライド等、他アプリケーションでの表示・編集互換性は対象外(2026-07-05 Clarifications 参照)。
