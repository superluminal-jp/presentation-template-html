# Feature Specification: コンポーネント集の付録一本化と PPTX 反映

**Feature Branch**: `007-appendix-consolidation`

**Created**: 2026-07-11

**Status**: Draft

**Input**: User description: "コンポーネント集ページ(components.html)を廃止し、その全コンポーネント実例をデッキの付録スライドに一本化する。合わせて PPTX エクスポート(scripts/pptx)にも付録スライドを含める。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 全コンポーネントを付録で一望する (Priority: P1)

デッキ利用者は、独立した別ページを開かずに、デッキ本体(index.html)の付録スライドだけで全コンポーネントの体裁と使い方を確認できる。

**Why this priority**: 本フィーチャーの中核価値。コンポーネント実例が「コンポーネント集」と「付録」の二箇所に分散している状態を解消し、単一情報源にする。これが成立すれば独立ページを廃止できる。

**Independent Test**: index.html(および生成された slides/)だけを閲覧し、対象の全コンポーネントが 1 回以上、体裁が判別できる形で実演されていることを確認できる。

**Acceptance Scenarios**:

1. **Given** デッキの付録スライド群、**When** 付録を通覧する、**Then** 自作 14 種(コールアウト/バッジ/タグ・チップ/ボタン/データテーブル/プログレスバー/統計(KPI)/凡例/区切り/チェックリスト/引用ブロック/キーバリュー/バナー(通知・緊急)/ステップナビゲーション)と DADS 公式 6 種(カード/箇条書きリスト/チェックボックス/ラジオボタン/プログレスインジケーター/パンくずリスト)の計 20 種すべてが 1 回以上出現する。
2. **Given** 付録内の各コンポーネント、**When** 表示を確認する、**Then** 色・枠・状態などの体裁がトークン準拠で崩れなく描画され、重複実演は最小限である。

---

### User Story 2 - 独立ページの廃止(リンク切れなし) (Priority: P2)

保守者は、廃止された「コンポーネント集」への参照が残っていない状態でリポジトリを保守できる。

**Why this priority**: 二重管理の温床を除去し、保守対象を 1 つに集約する。P1(付録に一本化)の完了が前提。

**Independent Test**: `components.html` が存在せず、README・ナビゲーション・ビルドスクリプト・テストのいずれからも同ファイルへの参照が無いことを検索で確認できる。

**Acceptance Scenarios**:

1. **Given** リポジトリ、**When** `components.html` への参照を全文検索する、**Then** ソース・ドキュメント・テストにヒットが 0 件である。
2. **Given** 既存の検証スイート、**When** 一式を実行する、**Then** 参照除去後もすべて緑のままである(コンポーネント CSS・トークン・vendor 資産は残置)。

---

### User Story 3 - PPTX にも付録を含める (Priority: P3)

PPTX 利用者は、エクスポートした .pptx を開くだけで付録のコンポーネント実例を確認できる。

**Why this priority**: HTML と PPTX の内容差を縮める。P1 の付録定義が確定してから反映するのが自然。

**Independent Test**: `npm run build:pptx` で生成した .pptx を開き、付録スライドが存在し、各要素がネイティブに編集可能(ラスタ画像でない)ことを確認できる。

**Acceptance Scenarios**:

1. **Given** PPTX エクスポート、**When** 生成物を開く、**Then** 付録スライドが 1 枚以上含まれ、コンポーネントの体裁(色・枠・状態)が判別できる。
2. **Given** 付録スライド上の各要素、**When** 選択する、**Then** テキストボックス・図形・ネイティブ表として編集でき、本文領域にラスタ画像が使われていない。

---

### Edge Cases

- デッキ本文が主張する「コンポーネント点数」などの数値表記が、一本化後の実体(20 種)と食い違わないこと。数値を掲げる箇所は実体に整合させる。
- 付録スライドが増えることでネイティブ 1280×720 サイズの見切れ(オーバーフロー)を起こさないこと。
- 生成物 slides/*.html が index.html と乖離しないこと(再生成で整合)。
- SVG チャート等「凝った」表現は PPTX では簡略化・省略しても、コンポーネントの識別性が保たれること。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、対象コンポーネント 20 種すべてをデッキの付録スライド上で 1 回以上実演しなければならない。
- **FR-002**: システムは、既存の付録スライド(A〜E 相当)と統合し、各コンポーネントを原則 1 回のみ実演しなければならない(意図的な再利用文脈を除き、同一コンポーネントの重複実演は 0)。
- **FR-002a**: バナー(通知・緊急)およびステップナビゲーションは、DS のパターンに倣った自作コンポーネント(トークン準拠)として追加する。DADS 公式コンポーネントの vendor 同期(`sync:dads`)は本フィーチャーでは拡張しない。
- **FR-003**: 独立した「コンポーネント集」ページ(`components.html`)は廃止(削除)されなければならない。
- **FR-004**: システムは、README・ナビゲーション・ビルドスクリプト・テストから `components.html` への参照を残してはならない(リンク切れ 0)。
- **FR-005**: コンポーネント用スタイル(`components.css` / `dads-components.css`)、デザイントークン、vendor 資産(`tokens/vendor`, `vendor/dads-components`)は残置され、付録スライドから利用可能でなければならない。
- **FR-006**: 付録スライド上のすべての表現はデザイントークンに準拠し、ブランド値のハードコードを含んではならない。
- **FR-007**: 付録スライドを含むデッキ全体が WCAG 2.2 AA のコントラスト基準を満たさなければならない。
- **FR-008**: PPTX エクスポートは、付録スライドを 1 枚以上含め、各要素をネイティブ要素(テキストボックス・背景・単純図形・ネイティブ表)で表現しなければならない(本文領域にラスタ画像を用いない)。
- **FR-009**: 正典は `index.html` とし、`slides/*.html` は生成物として再生成により整合が保たれなければならない(手編集しない)。
- **FR-010**: デッキ本文中のコンポーネント点数等の数値表記は、一本化後の実体に整合していなければならない。
- **FR-011**: practices の網羅性(coverage)・相互参照(crossrefs)チェックは、一本化後も緑を維持しなければならない。

### Key Entities *(include if feature involves data)*

- **付録スライド**: コンポーネントの実例を提示するデッキ末尾のスライド群。各コンポーネントの体裁・状態・使い方の見本を含む。
- **コンポーネント(実演対象)**: 自作 14 種 + DADS 公式 6 種の計 20 種。各々が名称・体裁・状態を持つ。
- **PPTX 付録スライド**: HTML 付録に対応するネイティブ PPTX スライド。テキストボックス・図形・表で構成。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 対象コンポーネント 20 種すべてが、デッキ付録で 1 回以上実演されている(欠落 0)。
- **SC-002**: リポジトリ全体で `components.html` への参照ヒットが 0 件で、同ファイルが存在しない。
- **SC-003**: 既存の検証スイート(トークン lint・crossrefs・coverage・slides 整合・visual・a11y)が一式緑で通過する(視覚回帰ベースラインは意図変更として更新後)。
- **SC-004**: `npm run build:pptx` で生成した .pptx に付録スライドが 1 枚以上含まれ、付録スライド上の要素にラスタ画像が 0 件(すべてネイティブ編集可能)。
- **SC-005**: デッキ本文のコンポーネント点数表記が実体(20 種)と一致している。

## Assumptions

- 「廃止」はファイル削除を意味する(利用者選択済み)。コンポーネント CSS と vendor 資産は付録が利用するため残置する。
- 既存の付録 A〜E の枠組みを活かしつつ、未実演コンポーネント(区切り、DADS 公式 6 種、および新規自作のバナー・ステップナビゲーション)を追加する形で拡張する。必要に応じて付録スライドを増設する。
- バナー(通知・緊急)とステップナビゲーションは、DS の同名パターン(design-system-example-components-html の notification/emergency-banner・step-navigation)に着想を得た自作コンポーネント。効果的なスライド作成(全幅の告知帯・手順の道標)に資するため本フィーチャーで追加する。政策ダッシュボード資産(policy-dashboard-assets)は Power BI テンプレート/地図データであり HTML 部品ではないため対象外。
- PPTX の付録は体裁(色・枠・状態)が伝われば良く、SVG 多系列チャート等の高度表現は簡略化・省略してよい(利用者許容済み)。
- 視覚回帰(visual)のベースラインは、付録・数値表記の意図的変更に合わせて更新する。
- ネイティブサイズ検証を壊さないため present モードは opt-in(既定 review)のままとする。
